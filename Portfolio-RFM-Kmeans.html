RFM Analysis & K-Means
Customer Segmentation

Combining rule-based lifecycle segmentation with unsupervised machine learning to build precision-targeted marketing strategies on Google's The Look E-Commerce dataset.

Google BigQuery BigQuery ML · K-Means Standard SQL The Look E-Commerce
01
Executive Summary

This project demonstrates how structured SQL analysis can be layered with unsupervised machine learning to transform raw transactional data into actionable customer intelligence. By applying RFM (Recency, Frequency, Monetary) scoring and K-Means clustering in BigQuery, the analysis produces a dual-lens segmentation framework capable of guiding real-world marketing decisions.

The work was originally conceived to solve a real business problem: a client's database was so skewed that meaningful segmentation was nearly impossible — 85% of customers had never reordered, and 57% had purchased only once. Rather than deliver an analysis built on insufficient data, the project was rebuilt on a richer public dataset to validate the methodology at scale.

Output

A production-ready analytics pipeline that segments customers into 11 behavioral groups, validates those groups using a 5-cluster ML model, surfaces high-value sub-segment outliers, and supports automated daily refresh via BigQuery scheduled queries.

02
Business Context & Problem Statement

Customer segmentation is one of the highest-leverage activities in marketing analytics. Sending the same message to every customer is both costly and ineffective — the core challenge is identifying who your customers are behaviorally, not just demographically, so that marketing spend is allocated where it is most likely to generate returns.

This project addresses three specific questions:

  • Which customers are at risk of churning and worth prioritizing for retention spend?
  • Which new customers show the most potential for conversion into loyal, high-value buyers?
  • Within each behavioral segment, are there sub-groups that require a differentiated offer rather than a one-size-fits-all campaign?

Why The Look E-Commerce? The original project used a client's proprietary data, but 85% of their customers had never reordered and 57% had purchased only once. This extreme skew made meaningful segmentation statistically unreliable. The Look provides a balanced, large-scale dataset that properly stress-tests the methodology — and using it was the right call.

03
Methodology
Layer 1 — RFM Scoring Engine

Each customer is scored 1–5 across three behavioral dimensions using NTILE quintiles. The key design decision was retaining raw values (actual dates and dollar amounts) alongside the scores, so that downstream ML output can always be translated back into real-world business metrics.

MetricDefinitionScoring Direction
Recency (R)Most recent order dateASC — more recent = score 5
Frequency (F)Count of distinct ordersDESC — more orders = score 5
Monetary (M)Sum of sale prices paidDESC — higher spend = score 5

RFM Scoring View

Calculates raw metrics and applies NTILE(5) quintile scores per user.

CREATE OR REPLACE VIEW `ecommerce-store-489019.client_analysis.RFM_scores` AS (
  SELECT *,
    NTILE(5) OVER (ORDER BY raw_recency ASC)    AS R_Score,
    NTILE(5) OVER (ORDER BY raw_frequency DESC) AS F_Score,
    NTILE(5) OVER (ORDER BY raw_monetary DESC)  AS M_Score
  FROM (
    SELECT user_id,
      MAX(created_at)                    AS raw_recency,
      COUNT(DISTINCT order_id)            AS raw_frequency,
      SUM(sale_price)                    AS raw_monetary
    FROM `ecommerce-store-489019.client_analysis.orders`
    WHERE status NOT LIKE 'Cancelled'  -- Exclude cancelled orders
    GROUP BY user_id
  )
);
Layer 2 — Rule-Based Segment Logic

Scored customers are classified into 11 named lifecycle segments using a CASE statement hierarchy. Segment order matters: Champions are evaluated before Loyalists to prevent higher-value customers from being downgraded by a less-specific rule.

LoyalistsR≥3, F≥3, M≥3Repeat buyers with strong brand trustNew arrivals, exclusive content

Layer 3 — K-Means Clustering (BigQuery ML)

To complement the rule-based system, a K-Means model was trained in BigQuery ML using R_Score, F_Score, and M_Score as features — and only those three columns. Including categorical fields like segment labels would contaminate unsupervised learning; any encoding would introduce artificial bias. KMEANS++ initialization was used to improve convergence by spreading initial centroids across the feature space.

K-Means Model Training

5-cluster model trained on RFM scores using KMEANS++ initialization.

CREATE OR REPLACE MODEL `client_analysis.RFM_clusters`
OPTIONS (
  MODEL_TYPE          = 'KMEANS',
  NUM_CLUSTERS        = 5,
  KMEANS_INIT_METHOD  = 'KMEANS++'
) AS
SELECT R_Score, F_Score, M_Score  -- Behavioral features only
FROM `client_analysis.RFM_scores_table`;
Layer 4 — Cross-Validation & Business Translation

The final step joins ML.PREDICT output back to the rule-based segments and replaces score-level data with real-world metrics — days since last purchase, average order value, and lifetime revenue — so findings are immediately usable by a marketing or leadership team.

Boardroom Analysis Query

Maps ML clusters to manual segments with business metrics (days & dollars).

WITH final_combined AS (
  SELECT
    s.customer_segment,
    p.CENTROID_ID                                   AS ai_cluster,
    st.raw_recency, st.raw_frequency, st.raw_monetary,
    st.R_Score, st.F_Score, st.M_Score
  FROM ML.PREDICT(MODEL `client_analysis.RFM_clusters`,
    (SELECT * FROM `client_analysis.RFM_scores_table`)) p
  JOIN `client_analysis.RFM_segments` s      ON p.user_id = s.user_id
  JOIN `client_analysis.RFM_scores_table` st ON p.user_id = st.user_id
)
SELECT
  customer_segment, ai_cluster,
  COUNT(*) AS customer_count,
  ROUND(AVG(DATE_DIFF(CURRENT_DATE(), EXTRACT(DATE FROM raw_recency), DAY)), 0)
                                                    AS avg_days_since_last_purchase,
  ROUND(AVG(raw_monetary), 2)                   AS avg_lifetime_revenue,
  ROUND(SUM(raw_monetary) / NULLIF(SUM(raw_frequency),0), 2)
                                                    AS avg_order_value
FROM final_combined
GROUP BY 1, 2
ORDER BY customer_segment, ai_cluster;

Analytical insight — C1 vs. C2

Cluster 1 and Cluster 2 are nearly identical on frequency (diff: 0.11) and monetary (diff: 0.16) — yet recency separates them by 2.58 points. Recency is the sole anchor differentiating these two groups, meaning a targeted re-engagement based on time-since-purchase is the lever, not an offer tied to spend amount.

Business impact

Without the ML layer, all ~57,000 Hibernating customers would receive the same campaign. With cluster cross-validation, the ~5,100 Cluster 4 Hibernating customers — with an AOV 2× higher than C5 — can receive a premium win-back while the rest go to a low-cost reactivation flow. This alone can meaningfully improve campaign ROI without increasing total spend.

05
Marketing Strategy Recommendations

Each recommendation is tied directly to the data and designed to match offer intensity to behavioral value — avoiding over-investment in low-probability segments and under-investment in high-return ones.

Champions High

Reward with early product access, loyalty tiers, or referral incentives. Low cost to retain — but high LTV risk if neglected.

Cannot Lose Them Critical

Immediate, personalized 1:1 outreach. F=5 and M=5 with dropping recency — every day of delay increases churn probability.

At Risk High

Win-back campaign with time-sensitive discount. Focus paid spend on C4 sub-cluster (AOV ~$99) — highest expected return per dollar.

Potential Loyalists (C4) High

Introduce loyalty program benefits now. $136 AOV and recent activity — this is the highest-value conversion opportunity in the database.

New Customers (C4) Medium-High

Launch a personalized post-purchase nurture series. $124 first-order AOV signals willingness to spend — build the habit while momentum is fresh.

Hibernating (C5) & Lost Low

Minimal budget. Test a low-cost email reactivation. No response → move to suppression list. Preserves deliverability and keeps CAC low.

06
Data Pipeline & Automation

A static segmentation is only useful once. The pipeline is designed to update automatically as new orders arrive, keeping segments current for ongoing campaign decisions.

LayerObjectTypeRole
1RFM_scoresViewRecalculates R, F, M on every execution
2RFM_segmentsViewApplies CASE-based segment logic; depends on RFM_scores
3RFM_final_tableScheduled TableMaterializes results daily at 08:00 UTC; stores history
Design note

Only the RFM_segments query needs to be scheduled. Because it depends on RFM_scores, scheduling segments automatically triggers scores to recalculate first. The processed_at timestamp column added to the scheduled query provides two things: freshness transparency in dashboards, and a silent failure detector — if the scheduled job stops running, the timestamp reveals exactly when data stopped updating.

Scheduled Refresh Query

Runs daily — overwrites RFM_final_table with current segment data and a freshness timestamp.

SELECT
  CURRENT_TIMESTAMP()  AS processed_at,  -- Pipeline health indicator
  *,
  CONCAT(R_Score, '-', F_Score, '-', M_Score) AS rfm_points,
  CASE
    WHEN R_Score >= 4 AND F_Score >= 4 AND M_Score >= 4 THEN 'Champions'
    WHEN R_Score >= 3 AND F_Score >= 3 AND M_Score >= 3 THEN 'Loyalists'
    WHEN R_Score <= 2 AND F_Score  = 5 AND M_Score  = 5 THEN 'Cannot Lose Them'
    WHEN R_Score <= 2 AND F_Score >= 3 AND M_Score >=  4 THEN 'At Risk'
    -- ... additional segment rules
    ELSE 'Lost'
  END AS customer_segment
FROM `client_analysis.RFM_scores`;
