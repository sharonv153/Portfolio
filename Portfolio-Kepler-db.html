Portfolio Project: Kepler Online Learning Relational Database Design

Tech Stack: Microsoft SQL Server, T-SQL, SQL Server Management Studio (SSMS)


Executive Summary

Kepler is a subscription-based EdTech platform designed for independent creators. The business requires a high-integrity relational database to manage complex revenue distribution, enrollment constraints (3-course maximum), and multi-tiered user activity tracking. This project demonstrates the transition from business requirements to a fully normalized Third Normal Form (3NF) schema, complete with complex business logic handled via T-SQL.


The Business Logic Challenge

The project required implementing specific constraints directly into the database logic:

  • Revenue Splitting: A fixed $20 subscription fee where instructors receive $5 per active student, and the platform retains the remainder.
  • State Management:Tracking "Active" status based on concurrent subscription payments and enrollment activity.
  • Capacity Constraints:Rigid enforcement of enrollment limits (max 3 courses) to maintain platform performance and educational focus.

Data Architecture (ERD)

The schema was designed to eliminate redundancy through strategic normalization. Key architectural decisions include:

  • Composite Key Implementation:Used in junction tables like Class_enrollment to manage many-to-many relationships.
  • Integrity Constraints:Extensive use of FOREIGN KEY constraints and BIT data types for Boolean logic (e.g., passed or completed status) to ensure data durability.

Advanced T-SQL Implementation

To demonstrate professional-grade SQL skills, the project focuses on set-based logic rather than procedural loops.

A. Strategic Revenue Analysis

Goal: Identify platform-heavy revenue streams where students are enrolled in one or fewer courses.

SELECT 
                  student_id, 
                  DATENAME(month, activity_date) AS active_month, 
                  (20 - (COUNT(course_id) * 5)) AS platform_net_revenue
              FROM Online_Courses.db_datawriter.Student_Activity
              GROUP BY student_id, DATENAME(month, activity_date)
              HAVING (20 - (COUNT(course_id) * 5)) > 5;

B. Data Aggregation & Virtualization

Goal: Create a Virtual Gradebook using SQL Views to centralize student performance across assignments, exams, and module completions.

CREATE VIEW db_datawriter.v_StudentGrades
              AS (
                  SELECT 
                      a.student_id, 
                      mo.course_id,
                      (SUM(a.scored_points) + SUM(e.scored_points) + SUM(m.total_points_earned)) AS cumulative_score
                  FROM Online_Courses.db_datawriter.Assignment_submission a
                  INNER JOIN Online_Courses.db_datawriter.Exam_submission e ON a.student_id = e.student_id
                  INNER JOIN Online_Courses.db_datawriter.Module_completion m ON e.student_id = m.student_id
                  INNER JOIN Online_Courses.db_datawriter.Module mo ON mo.module_id = m.module_id
                  GROUP BY a.student_id, mo.course_id
              );

C. The "Creator Payout" Financial Report

This query identifies "Active" students from the `Student_Activity` table for a specific month (January) and joins them with the `Course` and `CourseCreator` tables to calculate the $5-per-student payout.

/* Requirement: Creators receive payout on the 5th for the previous month.
                  Logic: Find all 'Active' status records for January 2026 and calculate $5 per student.
                  */
                  DECLARE @PayoutMonth INT = 1; -- January
                  DECLARE @PayoutYear INT = 2026;
                  SELECT 
                      CC.CreatorID,
                      CC.FirstName + ' ' + CC.LastName AS CreatorName,
                      C.CourseTitle,
                      COUNT(SA.StudentID) AS ActiveStudentCount,
                      COUNT(SA.StudentID) * 5.00 AS TotalPayoutAmount
                  FROM CourseCreator CC
                  JOIN Course C ON CC.CreatorID = C.CreatorID
                  JOIN StudentActivity SA ON C.CourseID = SA.CourseID
                  WHERE SA.StudentCourseStatus = 'Active'
                    AND MONTH(SA.ActivityDate) = @PayoutMonth
                    AND YEAR(SA.ActivityDate) = @PayoutYear
                  GROUP BY CC.CreatorID, CC.FirstName, CC.LastName, C.CourseTitle
                  ORDER BY TotalPayoutAmount DESC;

Future Scalability

  • Schema Extensibility: Mention that the modular design (separating Course, Module, and Assignment) allows Kepler to easily add new content types, such as live workshops or peer-reviewed projects, without refactoring the core architecture.
  • Performance Optimization: The next phase would involve implementing Stored Procedures for automated monthly invoicing and SQL agent jobs to process creator payouts on the 5th of every month.
